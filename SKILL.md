# Content Update

## 基本流程

1. 先读取当前内容、更新目标和相关上下文。
2. 先做差分分析，再进入方案整理。
3. 按固定顺序输出：
   - 差分
   - 现状
   - 实装后的预想
   - 如何实装
   - 具体的步骤
   - 关联修正检查
4. 如果已经落地修改，再检查与已修正内容关联的代码、数据、测试和文档。

## 差分分析

- 先明确当前内容是什么。
- 再明确更新后的目标内容是什么。
- 优先按“新增 / 删除 / 调整”整理差分。
- 如果用户没有给出完整的新内容，就根据需求推导目标状态，并明确写出假设。
- 如果差分涉及规则、行为或结构变化，不要只写文字变化，也要写功能层影响。

可使用这样的差分表达：

```md
## 差分

- 新增：
- 删除：
- 调整：
- 影响范围：
```

## 固定整理顺序

在差分之后，严格按以下顺序整理：

### 现状

- 概括当前内容、当前行为或当前结构。
- 说明当前存在的问题、限制或与目标不一致的地方。

### 实装后的预想

- 描述更新完成后的理想状态。
- 说明用户最终会看到什么变化。
- 如果涉及代码或数据行为，也要说明预期行为变化。

### 如何实装

- 先拆分影响层级：文档、UI、数据、规则、测试、构建产物。
- 说明应该改哪些文件、哪些模块、哪些描述。
- 优先保持模块边界清晰，不要把多种职责混在一次修改里。

### 具体的步骤

- 给出可以直接执行的顺序步骤。
- 步骤要从确认上下文开始，到改动、验证、同步文档结束。
- 如果任务跨文件，优先从源头定义文件开始，再到调用方和文档。

推荐模板：

```md
## 现状

## 实装后的预想

## 如何实装

## 具体的步骤
1. 
2. 
3. 
```

## 关联修正检查

在完成内容更新后，最后追加一次关联检查，至少覆盖以下方向：

- 是否有被改动内容引用到的代码、配置、类型或数据表仍保持旧状态。
- 是否有与本次修改相关的 UI 文案、提示文字、规则说明或注释需要同步。
- 是否有测试、样例数据、快照、fixture 或构建产物需要同步更新。
- 是否有导入路径、文件名、函数名、字段名或文档链接因为本次改动而失效。

## 针对本仓库的检查点

如果内容更新发生在这个仓库，优先检查：

- 文档与协作规则：
  - `AGENT.md`
  - `SKILL.md`
  - `memory-bank/architecture.md`
  - `memory-bank/progress.md`

- 游戏内容与规则：
  - `src/data/`
  - `src/engine/`
  - `tests/engine.test.js`

- UI 文案与展示：
  - `src/App.tsx`
  - `src/components/`

- 运行与产物同步：
  - `dist/`
  - `index.html`

## 落地修改时的默认要求

- 改代码前先看 `memory-bank/architecture.md` 和 `memory-bank/progress.md`。
- 改代码后把重要变化追加到 `memory-bank/progress.md`；如果本次修改只涉及 `.md` 文档文件，则不需要写入 `memory-bank/progress.md`。
- 所有 `.md` 文档文件必须使用中文书写。
- 如果项目结构、文件职责或工作流变了，也同步更新 `memory-bank/architecture.md`。
- 如果行为发生变化，检查是否需要运行构建或测试。
- 当有修改使魔以及追加使魔的时候，请同步更新使魔图鉴文档；默认目标为 `design/minion.md`，若当前仓库实际维护的是 `design/minion_codex.md`，则必须同步更新该文件并保持内容一致。
- 内容更新时，必须同步检查并更新 `design/game_design.md`。
- 规则更新时，必须同步检查并更新 `design/game_rule.md`。
